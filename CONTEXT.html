Domain Vocabulary
Work & Orchestration
  • Task Unit of work mirrored from an External Tracker and processed by Talby. avoid: internal ticket, local task
  • Task ID Talby-generated identifier unique inside Talby. Not the tracker's key. avoid: external issue id, tracker key
  • Workspace Policy boundary that defines Stage Vocabulary, Eligibility Rules, and governed external sources. May govern multiple trackers. avoid: integration connection, single board
  • External Tracker Third-party system that owns the canonical task lifecycle. avoid: Talby backlog, internal board
  • Source Reference Integration-managed mapping from a Task to its originating record in an External Tracker. avoid: primary task id, public task id
  • Stage Normalized external workflow position of a Task, used for routing and handoff. avoid: execution state, internal runtime progress
  • Stage Vocabulary Talby-configured set of valid Stages for a specific workspace or integration context. avoid: global workflow taxonomy, universal stages
  • Eligibility Rule Configuration rule determining whether a Task is eligible for Talby to start a Workflow Run. avoid: task configuration, implicit pickup rule
  • Execution State Talby-managed progress metadata about how agents are handling a Task. avoid: task status, tracker status
  • Tracker Projection Outbound update Talby writes into the External Tracker to reflect work performed in Talby (stage changes, comments, summaries, Claim Markers). avoid: mirrored field, shared ownership
Claims & Execution
  • Claim Talby's exclusive right for a Workflow Run to work a Task for a bounded period. Only a Workflow Run may hold one. avoid: pickup, lock
  • Lease Time-bounded validity window of a Claim; must be renewed to retain ownership. avoid: permanent assignment, reservation
  • Claim Marker Active Claim's representation in the External Tracker: assignee, title prefix, label, or status. avoid: claim itself, internal lease state
  • Claim Outcome Fixed vocabulary for why a Claim ended: completed · handed_off · abandoned · expired · revoked · failed. avoid: generic result, free-form status note
  • Handoff Projecting a Task into a new external stage so the next worker or human can take responsibility. Normally releases the active Claim. avoid: silent status change, internal retry
  • Workflow Orchestration definition structuring how work is carried out across one or more steps. avoid: worker, claim owner
  • Workflow Run One execution instance of a Workflow for a specific Task or Subtask. Owns the Claim while actively working. avoid: worker pool, workflow template
  • Workflow Run Outcome Business-level result: completed · handed_off · failed · canceled. avoid: infrastructure status, free-form result string
  • Child Workflow Invocation Execution-internal child workflow launched by a Workflow Run to do bounded work and return a result. No own Claim, Conversation, or Workflow Run. avoid: subtask, child task, independent workflow run
  • Subtask Talby-internal unit of work derived from a Task, coordinated under that Task's active Claim. avoid: external child issue, second task claim
  • Worker Talby infrastructure runtime that hosts execution but does not own task responsibility in the domain model. avoid: claim owner, business actor
Agent System
  • Skill Instruction folder that teaches Talby how to perform specific work. May be available during execution and may also be directly invocable when its metadata allows it. avoid: tool configuration, persona
  • Tool Configured integration (MCP or CLI) plus usage guidance. Declared directly on an Agent. avoid: skill, persona
  • Agent Persona that declares its own Tools and may work with Skills made available by Workflow configuration. avoid: runtime owner, skill bundle
  • Agent Instance Runtime participation of an Agent spawned by a Workflow Run within a Task Conversation. Subtype of Participant. avoid: agent definition, worker
  • Prompt Versioned directed command for a specific Agent with a narrower goal than general invocation. Declares the Agent it targets. avoid: persona, workflow run, ad hoc runtime instruction
  • Evaluation Defined assessment measuring quality or fitness of a Skill, Agent, or Prompt. avoid: ad hoc opinion, runtime outcome
  • Agent Repository Dedicated repository storing Talby's Agents, Prompts, Skills, Tools, and Evaluations. avoid: write repository, task repository
  • Custom Agent Repository Separate repository for workspace-specific Agent definitions that differ from the globally available set. avoid: workspace override inside the shared agent repository
People & Operations
  • Operator Human user who supervises Talby work, intervenes when needed, and manages task flow. Subtype of Participant. Human in v1. avoid: end customer, passive observer
  • Participant Conversation participant identity that contributes to or queries a Conversation. avoid: anonymous sender, worker session
  • Attention Item User-facing signal that Talby requires human intervention or acknowledgment for a Task. May pause the Workflow Run (Claim held) or end it (Claim released). avoid: low-level notification delivery, silent error
  • Attention Queue Operator's primary working surface in the v1 UI — list of open Attention Items. avoid: passive dashboard, generic task list
  • Operator Action Set Minimal v1 actions: post to Conversation, resolve/reopen Attention Item, trigger Handoff, invoke Prompt, invoke Agent with free-form text, open PR/Repo/Tracker links. avoid: full administration surface, generic edit menu
  • Conversation Task-scoped collaborative thread of messages and memory updates involving Participants. Primary panel in the v1 task view. Begins at task ingestion; may reopen if task reactivates. avoid: feature-wide log, finished-only session
Git Delivery
  • Repository Source repository Talby may read from or write to while executing work. avoid: workspace, tracker source
  • Repository Profile Talby-managed metadata attached to a Repository: purpose, read/write role, routing cues. Lives inside and versioned with the repo. avoid: ad hoc repo note, workflow run state
  • Repository Binding Talby policy association assigning a Repository to one Workspace by default. Not absolute Git-level access control. avoid: hard access control, exclusive Git permission
  • Repository Target Use of a Repository as part of work for a Task. A Task may read from many Repository Targets. avoid: separate task, workspace binding
  • Write Repository Single Repository where Talby opens the primary Pull Request for a Task in v1. Selected during routing before Claim acquisition. avoid: read-only repository, workspace
  • Change Branch Writable branch Talby uses to produce the primary Pull Request for a Task. Strategy selected by the router before execution starts. avoid: repository profile, task id
  • Pull Request Proposed change on one Write Repository. Primary git delivery unit for a Task in v1. May be opened early and updated incrementally. Stays with the Task across serial Workflow Runs and handoffs. avoid: task, branch set, multi-repo bundle
Memory System
  • Memory Thread Canonical external memory-system thread storing records of one Conversation. One-to-one with Conversation in v1. avoid: secondary shard, parallel conversation thread
  • Memory Record Stored fact, event, artifact, or message fragment in a Conversation. Includes issuing Participant identity. Append-only in v1. avoid: raw chat dump, detached log line
  • Memory Feed Baseline set of events Talby ingests automatically: task ingestion, comments, Participant messages, claim events, handoff events, artifact links. avoid: full system exhaust, manual-only save
  • Memory Query Request to the memory system scoped to a Participant, a Conversation, and a Workspace. avoid: global unscoped search, anonymous lookup
  • Organization Memory Memory available across the broader Talby system beyond a single Conversation (when the external memory system allows it). avoid: task-local chat history, unrestricted dump
  • Artifact Summary Extracted description or metadata representing an artifact in memory without storing full raw contents. avoid: full binary payload, entire raw log
  • First-Hand Record Memory Record emitted by Talby from observed messages, events, and artifacts — without adding new inferred facts. LLM-assisted compression is allowed; full per-input provenance not required in v1. avoid: inferred conclusion, synthetic organizational insight
Key v1 Constraints
A Task has at most one active Workflow Run at a time.
A Task has at most one active Claim at a time.
Only a Workflow Run may hold a Claim. Humans receive work through Handoff in the External Tracker.
A Task has at most one primary Pull Request on one Write Repository in v1.
The Write Repository and Change Branch strategy are selected during routing — before the Claim is acquired and the Workflow Run starts.
The primary Pull Request stays with the Task across serial Workflow Runs and handoffs by default.
A Conversation maps to exactly one canonical Memory Thread in v1.
Memory Records are append-only in v1. Corrections are emitted as new records.
Talby does not auto-retry a failed Workflow Run. The run releases its Claim and creates at least one Attention Item.
A Child Workflow Invocation does not own a Claim, Conversation, or Workflow Run of its own.
Talby emits only First-Hand Records. Inferred conclusions are not emitted as Memory Records.
A Skill may be used by any Agent; no agent-specific compatibility metadata exists.
A directly invocable Skill used as a step's base instruction becomes part of the step's effective Skills set even if not previously listed.
Step overrides to Workflow defaults must use explicit replace or merge semantics — never implicit override.
Resolved Ambiguities
status→ Execution State for Talby progress · Stage for external workflow position
"status" was used to mean both tracker lifecycle and Talby progress metadata. Resolved: use Execution State for Talby-managed progress, and reserve task lifecycle for the External Tracker. Use Stage for normalized external workflow position.
assignment→ Claim for exclusive rights · Lease for the bounded window
"assignment" could imply a durable owner. Resolved: use Claim for exclusive execution rights and Lease for the time-bounded ownership window.
agent→ Agent for capability definition · Agent Instance for runtime participant
"agent" could mean both a reusable capability and a running executor. Resolved: use Agent for the capability definition and Agent Instance for the runtime participant.
subtask→ Subtask is Talby-internal only, under the active Claim
"subtask" could be mistaken for an external tracker child item. Resolved: use Subtask only for Talby-internal derived work under a Task's active Claim.
task id→ Task ID is Talby-generated · Source Reference maps to external
"task id" could mean either Talby's identifier or the tracker's. Resolved: use Task ID for the Talby-generated identifier and Source Reference for the external mapping.
stage change→ Handoff when a stage change should release the Claim
"stage change" could mean either continued work or transfer of responsibility. Resolved: use Handoff when the stage change is meant to release the current Claim so the next worker can take responsibility.
workspace→ A Workspace may govern multiple External Trackers
"workspace" could be mistaken for one tracker connection only. Resolved: a Workspace is a policy boundary; it may govern multiple External Trackers when they share the same policy boundary.
repository belongs to one workspace→ Repository Binding is default policy, not hard Git isolation
"repository belongs to one workspace" could be mistaken for absolute Git-level isolation. Resolved: Repository Binding is Talby's default policy ownership, not a guarantee that other workspaces cannot act at the Git hosting level.
multi-repo work→ In v1, one Task has at most one primary Pull Request on one Write Repository
"multi-repo work" could imply multi-repo writes by default. Resolved: in v1 a Task may read many repositories via Repository Targets, but it has at most one primary Pull Request on one Write Repository.
repository purpose→ Repository Profile inside each repo, versioned with it
"repository purpose" could drift into undocumented routing folklore. Resolved: use Repository Profile for Talby-managed metadata that drives read/write selection. It lives inside and is versioned with its Repository.
write repository selection→ Chosen during routing, before Claim acquisition
"write repository selection" could be mistaken for an in-run discovery step. Resolved: the Write Repository is chosen during routing before the Claim is acquired and the Workflow Run starts.
pull request ownership→ Primary PR stays with the Task across serial handoffs
"pull request ownership" could drift from task-level continuity to run-level churn. Resolved: the primary Pull Request stays with the Task across serial Workflow Runs and handoffs by default.
prompt→ Prompt is a versioned asset; free-form text is just invocation input
"prompt" could be mistaken for any ad hoc instruction. Resolved: Prompt means a versioned reusable asset in the Agent Repository; free-form runtime text is invocation input, not a Prompt domain object.
workspace-specific agents→ Agents are global by default; variants live in a Custom Agent Repository
"workspace-specific agents" could imply per-workspace toggles in the shared catalog. Resolved: Agents are globally available by default; workspace-specific variants live in a separate Custom Agent Repository.
responsibility→ Only Workflow Runs hold Claims; humans receive work via Handoff
"responsibility" could imply either Talby runtime ownership or external human assignment. Resolved: only Workflow Runs hold Claims; humans receive work through Handoff in the External Tracker.
skill availability→ A Skill may be available without being directly invocable
"skill availability" could be confused with explicit invocation. Resolved: a Skill may be available during execution while remaining non-invocable if its metadata says so. Directly invoking one Skill does not remove others from the step.
step override→ Must use explicit replace or merge semantics
"step override" could be interpreted inconsistently. Resolved: a step must use explicit replace or merge semantics when changing workflow-default Agents or Skills, rather than an ambiguous implicit override.
inferred summaries→ Talby emits only First-Hand Records; inferences are not emitted
"inferred summaries and conclusions" could be treated as memory records. Resolved: Talby emits only First-Hand Records. The memory system may derive new conclusions from them, but that inferred knowledge is not emitted by Talby as a record.
